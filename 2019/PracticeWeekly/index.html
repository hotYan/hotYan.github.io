第七周

JS自动插入分号的机制？

1、什么是Automatic Semicolon Insertion (ASI, 自动分号插入机制)　？

按照ECMAScript标准，一些特定语句（statement）必须以分号结尾，分号代表这段语句的终止。实际这些分号是可以省略的，当分号省略的情况下JavaScript引擎会判断语句该在哪里终止。这种行为被叫做”自动插入分号“，简称ASI，实际上分号没有被真的插入。

特定的语句有：

  • 空语句
  • let
  • const
  • import
  • export
  • 变量赋值
  • 表达式
  • debugger
  • continue
  • break
  • return
  • throw

其他语句有两种情况：

  • 不需要分号
  • 分号不能省略


2、JS如何解析代码

解析器在解析代码时，会把代码分成很多token（一个token相当于一小段有特定意义的语法片段）。当解析器在解析语法时会一个一个读入token尝试构成一个完整的语句（statement)，直到碰到特定情况（比如语法规定的终止）就会判断该语句在此处结束了。用token构成语法的过程中类似于正则里面的贪婪匹配，解析器总是试图用尽可能多的token构成语句。

任意token之间都可以插入个或多个换行符，不影响JS的解析。这个特性可以让开发者通过增加代码的可读性，更灵活地组织语言风格。我们平时写的跨多行的数组，字符串拼接，和链式调用都属于这一类。不过在省略分号的风格中，这种解析特性会导致一些意外情况(比如以 / 开头的正则会被理解成除法)。事实上这并不是省略分号的风格的错误，而是没有理解 JS 解释器的工作原理导致的。如果倾向省略分号的风格，那了解 ASI 是必修课。


3、ASI规则

ECMAScript 标准定义的 ASI 包括 三条规则 和 两条例外。

三条规则（难顶难顶~）

  1. 解析器从左往右解析代码（读入 token），当碰到一个不能构成合法语句的 token 时，它会在以下几种情况中在该 token 之前插入分号，此时这个不合群的 token 被称为 offending token ：

    • 如果这个 token 跟上一个 token 之间有至少一个换行。

    • 如果这个 token 是 }。

    • 如果 前一个 token 是 )，它会试图把前面的 token 理解成 do...while 语句并插入分号。

  2. 当解析到文件末尾发现语法还是有问题，就会在文件末尾插入分号。

  3. 当解析时碰到 restricted production 的语法（比如 return），并且在 restricted production 规定的 [no LineTerminator here] 的地方发现换行，那么换行的地方就会被插入分号。

两条列外（就算符合上述规则，如果分号会被解析成下面的样子，它也不能被自动插入）

  • 分号不能被解析成空语句
  • 分号不能被解析成for语句头的两个分号之一

第六周

如何突破浏览器并发个数？结合一些比较知名的网站，看看他们的网站是如何做类似优化的。

1、不同浏览器不同版本的并发限制是基于多方面因素考量出的优化结果：

  • 对客户端操作系统而言，过多的并发涉及到端口数量和线程切换开销。

  • HTTP/1.1有Keep Alive，支持复用现有连接，等请求返回回来后，再复用连接请求可以快很多。

  • 将所有请求一起发给服务器，也很可能会引发服务器的并发阈值控制而被BAN。

2、也并不是浏览器限制并发了，我们就没别的办法了。提升页面打开速度的技术有很多：

  • domain hash：对资源做哈希，请求到不同的域下面。

  • cookie free：前后端分离，减少不必要的cookie提交。

  • css sprite：将零星的图片整合到一张大图中，减少请求次数。

  • js/css combine：资源合并，减少请求次数，不过也会增加文件修改的几率。

  • max expires time：合理设置客户端缓存时间。

  • loading images on demand：图片按需加载。

3、域名发散与收敛

  • 域名发散：因为浏览器对同一域名下的最大连接数做了限制，为了让浏览器并发加载，将资源分散到不同域名下，缺点是需要DNS解析更多的域名。

  • 域名收敛：在移动场景下，为了解决DNS解析耗时问题，将资源汇总到一个或更少的域名下。

    4、一个TCP连接可以同时发送几个HTTP请求？

HTTP/1.1中，单个TCP连接，在同一时间只能处理一个http请求，虽然存在流水线（Pipelining）技术支持多个请求同时发送，但由于实践中存在很多问题无法解决，所以浏览器默认是关闭，所以可以认为是不支持同时多个请求。

HTTP2提供了多路传输功能，多个http请求，可以同时在同一个TCP连接中进行传输。

5、浏览器http请求的并发性是如何体现的？并发请求的数量有没有限制？

页面资源请求时，浏览器会同时和服务器建立多个TCP连接，在同一个TCP连接上顺序处理多个HTTP请求。所以浏览器的并发性就体现在可以建立多个TCP连接，来支持多个http同时请求。

Chrome浏览器最多允许对同一个域名Host建立6个TCP连接，不同的浏览器有所区别。

6、网页中的图片资源为什么分放在不同的域名下?

浏览器对并发请求的数目限制是针对域名的，即针对同一域名（包括二级域名）在同一时间支持的并发请求数量的限制。如果请求数目超出限制，则会阻塞。因此，网站中对一些静态资源，使用不同的一级域名，可以提升浏览器并行请求的数目，加速界面资源的获取速度。

什么是浏览器同源策略？我们在日常开发调试时有无跨域？如有，是如何跨域的？

1、同源策略&跨域

浏览器的同源策略导致了跨域。

2、同源策略

限制了从同一个源加载的文档或脚本如何与来自另一个源的资源进行交互。是一个用于隔离潜在恶意文件的重要安全机制,即保证用户信息安全，防止恶意的网站窃取数据，如跨站请求伪造(CSRF)攻击。

3、同源

  • 协议相同
  • 域名相同
  • 端口相同

4、非同源，三种行为受到限制

  • Cookie、LocalStorage、IndexDB 无法读取
  • DOM无法获得
  • AJAX请求不能发送

5、解决跨域

  • 跨域资源共享（Cross-Origin Resource Sharing）
  • JSONP
  • 服务器代理
  • document.admin

  • window.name
  • 图像Ping

5.1 CORS

浏览器将CORS请求分为：简单请求和非简单请求,同时满足一下两大条件就属于简单请求：

  • ① 请求是 HEAD、GET、POST 三种方法之一；
  • ② HTTP头部信息不包含除 Accept、Accept-Language、Content-Language、Last-Event_ID、Content-Type以外的字段

不满足则属于非简单请求；

5.2 JSONP

  • JSONP 跨域的基本原理:由于 script 标签不受浏览器同源策略的影响，允许跨域引用资源。因此可以通过动态创建 script 标签，然后利用 src 属性进行跨域。

第一周

请描述浏览器输入URL到页面展示的整个过程??

1、流程概览

  • 将域名解析成IP地址(DNS 解析)
    • 输入URL地址,浏览器从url中解析出主机名、端口号
    • 浏览器将主机名转换成服务器ip地址（浏览器先查找本地DNS缓存列表,没有的话,再向浏览器默认的DNS服务器发送查询请求,同时缓存）
  • 浏览器建立一条与目标Web服务器的TCP连接(TCP 三次握手)
  • 浏览器向服务器发送一条HTTP请求报文
  • 服务器处理请求并向浏览器返回一条HTTP响应报文
  • 浏览器解析渲染绘制页面,浏览器可能会重复发送http请求,直至资源全部加载完毕(浏览器渲染机制）
  • 断开连接(TCP 四次挥手)


2、DNS解析


3、浏览器渲染机制

  • 根据 HTML 解析出 DOM 树

    • 解析HTML构建成一个DOM树(Document Object Model,文档对象模型），DOM 树的构建过程是一个深度优先遍历过程：当前节点的所有子节点都构建好后才会去构建当前节点的下一个兄弟节点。

    • 若遇到 script 标签，则 DOM 树的构建会暂停，直至脚本执行完毕。

  • 根据 CSS 解析生成 CSS规则树(CSS Rule Tree)

    • 解析 CSS规则树 时 js执行 将暂停，直至 CSS规则树 就绪。

    • 浏览器在 CSS规则树 生成之前不会进行渲染。

  • 根据 DOM树 和 CSSOM 来构造 Rendering Tree（渲染树）

    • DOM树 和 CSS规则树 全部准备好了以后，浏览器才会开始构建 渲染树。

    • 精简CSS 并可以加快 CSS规则树的构建，从而加快页面相应速度。

      注意： Rendering Tree 渲染树并不等同于 DOM 树，因为一些像 Header 或 display:none 的东西就没必要放在渲染树中了。

  • 根据渲染树计算每一个节点的信息

    有了Render Tree，浏览器已经能知道网页中有哪些节点、各个节点的CSS定义以及他们的从属关系。下一步操作称之为Layout，顾名思义就是计算出每个节点在屏幕中的位置 layout render tree：

    • 布局（Layout）：通过渲染树中渲染对象的信息，计算出每一个渲染对象的位置和尺寸

    • 回流：在布局完成后，发现了某个部分发生了变化影响了布局，那就需要倒回去重新渲染。

  • 根据计算好的信息绘制页面

    绘制(Paint)，即遍历render树，并使用浏览器UI后端层绘制每个节点：

    • 绘制阶段，系统会遍历呈现树(render tree)，并调用呈现器的“paint”方法，将呈现器的内容显示在屏幕上。

    • 重绘：某个元素的背景颜色，文字颜色等，不影响元素周围或内部布局的属性，将只会引起浏览器的重绘。


4、 重绘 & 回流

重绘不一定回流，回流一定重绘。

  • 重绘： 不影响周围布局，修改元素背景颜色、文字颜色等行为。
  • 回流：元素尺寸发生了变化，需重新计算渲染树，重新渲染的行为。
